## 核心定义

**事实类问题（输出 1）**：存在唯一正确答案，任何客服都会给出相同回答。答案来源于平台产品文档、帮助中心、FAQ、规则政策、操作手册、功能说明。

**非事实类问题（输出 0）**：答案不唯一。需要结合投放经验、优化策略、客户具体数据/场景才能回答，不同优化师可能给出不同建议。

## 判定流程（按顺序执行，命中即停）

**Step 1 — 排除不可回答的问题**
问题是否属于以下任一情况？
- 必须查看客户**具体账户/订单/业务数据**才能回答（如"我的XX是否成功""帮我查XX"）
- 仅为**单个宽泛名词**，无明确诉求（如"保障""账户周报"）
- 指向**个案申请/处理**，无统一标准流程（如"申请技术核查"）
→ 是 → 输出 0

**Step 2 — 识别投放优化类问题**
问题是否在寻求**效果提升的策略或建议**？关键特征：
- 目标是提升/优化某个效果指标（转化率、点击率、激活率、曝光量、跑量、点击均价等）
- 询问效果层面的影响因素（"XX受什么影响""XX速度为什么慢"）
- 需要投放经验才能回答（素材方向、定向策略、出价技巧、案例方案）
- 隐含数据分析/调优诉求（"XX报告怎么看"暗示要分析数据）
→ 是 → 输出 0

**Step 3 — 识别事实类问题**
问题是否可从平台公开文档获得**统一的标准答案**？包括：
- 规则/政策/资费（赔付规则、税率、保障期、到账时间、扶持比例、消耗优先级）
- 功能说明/能力确认（是否支持、有无某功能、渠道/产品优缺点、术语定义、分类差异）
- 标准操作步骤（怎么配置、怎么开通、怎么搭建计划、怎么提现退款、怎么填写）
- 准入/合规/资质（是否有资格、行业限制、合规要求、备案方法）
- 通用故障排查（平台公开的标准排查流程，非个案诊断）
- 平台预设属性（官方赛道画像、内置形象/选项、行业分类标准）
→ 是 → 输出 1

## 三条核心判定原则

**原则一：看"答案是否唯一"，而非看"问题像不像优化"**
很多问题表面涉及投放词汇，但答案是确定的：
- 「怎么投私信获取线索」→ 标准操作步骤，答案唯一 → 1
- 「如何提升私信获取线索的转化率」→ 效果优化，答案不唯一 → 0
- 「视频的转化率不高应该修改哪个部分」→ 平台公开的通用排查指引，答案唯一 → 1
- 「我的视频转化率不高怎么优化」→ 需分析具体数据定制方案 → 0

**原则二：区分"平台规则层面"和"投放效果层面"**
同一个动作，问规则还是问效果，结论不同：
- 「主体变更对数据有影响吗」→ 问平台规则层面的影响 → 1
- 「换主体会不会影响跑量」→ 问投放效果层面的影响 → 0
- 「不同行业有不同影响吗」→ 问平台预设的行业规则差异 → 1
- 「千次曝光速度受什么影响」→ 问投放效果的影响因素 → 0

**原则三：省略句式默认问操作/规则，但有例外**
- 有明确功能/动作指向 →「视频号投放」→ 默认问怎么操作 → 1
- 两个概念并列 →「SaaS和传统软件区别」→ 默认问定义差异 → 1
- 单个宽泛名词无指向 →「保障」→ 诉求不明确 → 0
- 多种诉求可能且无明确指向 →「小程序备案」→ 诉求不明确 → 0

**Error Analysis:**
### 1. 每个错误案例的根本原因
#### 案例1：怎么才能提高通过手机号定向收到广告的概率
- 根本原因：问题中“提高”“概率”的表述触发了Step2“寻求效果提升策略”的表面特征判定，因Step2优先级高于Step3，被直接归类为非事实类输出0；但该问题实际询问的是平台公开的通用标准操作方法，答案唯一，属于事实类，当前规则未对“提升类的通用操作”和“提升类的个性化策略”做区分，导致误判。
#### 案例2：个人的话，能通过广告收益吗
- 根本原因：该问题本质是询问平台对个人主体参与广告收益的通用准入规则，有统一标准答案，属于事实类；但当前规则未明确区分“无个性化指代的通用规则询问”和“需结合个体情况的个案资格查询”的边界，被误判为需要结合个人具体场景才能回答的非事实类，导致输出0。

---
### 2. 当前prompt规则的通用性缺陷
1. **判定流程优先级不合理**：Step2（投放优化类判定）优先级高于Step3（事实类判定），大量表面带优化类关键词但本质是询问通用规则/操作的事实类问题，被提前拦截误判，没有进入Step3校验环节。
2. **特征判定缺乏排除机制**：Step2仅通过“提升指标、询问影响因素”等表面特征判定为非事实类，未排除“询问平台公开通用操作/固定规则”的场景，特征判定过于绝对，脱离“答案是否唯一”的核心标准。
3. **通用/个案问题边界模糊**：未明确“无个性化指代的通用问题”和“带个人/账户指代的个案问题”的判定差异，导致大量通用准入、功能确认类问题被误判为需要结合个体数据的非事实类。
4. **核心原则落地性不足**：“答案是否唯一”的核心判定原则仅作为说明，未嵌入各步骤的判定逻辑中，导致步骤特征判定和核心标准脱节。

---
### 3. 可泛化的规则改进建议
1. **调整核心判定优先级**：将“答案是否唯一且平台公开可查”作为全流程最高优先级判定标准，所有步骤的特征判定都需服从该标准：即使命中Step1、Step2的表面特征，只要答案唯一公开，就判定为事实类输出1。
2. **优化Step2的判定逻辑，增加排除校验**：命中Step2表面特征的问题，需额外校验是否属于以下排除场景，是则不命中Step2，进入Step3判定：①无“我的、我账户、我的案例”等个性化指代，询问的是实现某目标的通用标准操作步骤；②询问的是平台规则层面的固定影响/统一规则，而非实际投放的波动效果；③属于通用功能是否支持、准入规则类的是非问题。
3. **补充通用/个案问题前置判定规则**：在Step1前增加子步骤：若问题无明确个性化指代，默认属于询问通用规则/操作，不触发Step1中“需要具体数据、个案申请”的判定；只有带明确个性化指代的问题，才校验是否属于Step1的排除场景。
4. **扩展Step3的事实类覆盖说明**：明确“询问实现某目标的通用操作步骤，只要平台有公开统一的操作流程，无论目标是否涉及效果提升，都属于事实类”。

**Error Analysis:**
### 1. 错误案例失败的根本原因
模型误将query中「提高...概率」的表述识别为Step2定义的「寻求效果提升的个性化策略建议」类诉求，进而判定为0；但该query实际询问的是手机号定向触达的官方标准配置操作方法，答案可从平台公开的定向规则、操作手册中获取，是统一唯一的，完全符合事实类问题定义，预期输出为1。本质是模型未区分「提升类诉求的实现依赖平台标准化规则操作」和「提升类诉求的实现依赖个性化投放经验/用户数据」两类场景的边界。

---
### 2. 当前prompt规则存在的通用性缺陷
1. **Step2投放优化类判定逻辑存在漏洞**：仅将「目标为提升效果相关指标」作为核心判定特征，未设置例外排除规则，没有区分「提升目标可通过平台公开标准化操作/固定规则实现」和「提升目标需要结合个性化数据、投放经验定制调优」两类场景，导致前者被批量误判为非事实类。

3. **Step3事实类判定的引导性不足**：现有Step3对「标准操作步骤」的范畴说明未明确包含「达成某一明确功能目标的固定操作指引」，未提示这类操作类诉求不受表述中是否带优化类词汇的影响。
